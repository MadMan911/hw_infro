# Стратегии балансировки нагрузки

## Обзор

Балансировщик (`src/llm/balancer.py`) поддерживает три стратегии и два дополнительных механизма: circuit breaker и rate limiting.

Выбор стратегии задаётся через `BALANCING_STRATEGY` в `.env`:
```
BALANCING_STRATEGY=round_robin   # или weighted, latency_based
```

---

## Стратегии

### 1. Round Robin (`round_robin`)

Запросы распределяются по провайдерам последовательно, по кругу.

```
Запрос 1 → mock-llm-1
Запрос 2 → mock-llm-2
Запрос 3 → mock-llm-3
Запрос 4 → mock-llm-1
...
```

**Плюсы:** Простота, равномерное распределение количества запросов.

**Минусы:** Не учитывает разную скорость провайдеров. Медленный провайдер получает столько же запросов, что и быстрый.

**Когда использовать:** Провайдеры однородны по характеристикам (одинаковые модели, близкая латентность).

---

### 2. Weighted (`weighted`)

Провайдерам назначаются веса. Провайдер с весом 2.0 получает в 2 раза больше запросов, чем провайдер с весом 1.0.

Веса берутся из `ProviderRegistry` (`weight` поле). Если провайдер не зарегистрирован в реестре, используется значение из статического словаря `weights` или 1.0.

```
mock-llm-1 (weight=3.0) → ~60% запросов
mock-llm-2 (weight=2.0) → ~40% запросов
mock-llm-3 (weight=0.0) → исключён
```

**Плюсы:** Позволяет направлять больше трафика на более мощные/дешёвые провайдеры.

**Минусы:** Веса статичны — не адаптируются к реальной нагрузке в runtime.

**Когда использовать:** Провайдеры с разной пропускной способностью или ценой. Например: основной GPT-4o-mini с высоким весом + резервный Anthropic с низким весом.

---

### 3. Latency-Based (`latency_based`)

Балансировщик отслеживает среднюю латентность каждого провайдера (EMA, α=0.3) и направляет новый запрос к провайдеру с наименьшим скором:

```
score(provider) = avg_latency_ms / priority
```

Где `priority` берётся из `ProviderRegistry` (по умолчанию 1). Провайдер с `priority=2` получает вдвое лучший скор при одинаковой латентности.

```
mock-llm-1: avg=105ms, priority=1  → score=105
mock-llm-2: avg=210ms, priority=2  → score=105  (tie)
mock-llm-3: avg=320ms, priority=1  → score=320  (не выбирается)
```

При запуске все латентности равны 0 → первые запросы идут по round-robin-логике (min из нулей).

**Плюсы:** Автоматически адаптируется к реальной скорости провайдеров.

**Минусы:** Может создать "горячего" провайдера: самый быстрый получает все запросы, что может его перегрузить.

**Когда использовать:** Провайдеры с сильно различающейся латентностью, где важен p50 времени ответа.

## Circuit Breaker

Каждый провайдер имеет независимый circuit breaker:

```
CLOSED ──(3 ошибки подряд)──► OPEN ──(60 сек)──► HALF_OPEN ──(1 успех)──► CLOSED
```

| Состояние | Поведение |
|---|---|
| CLOSED | Нормальная работа |
| OPEN | Провайдер исключён из пула, запросы идут к другим |
| HALF_OPEN | Один пробный запрос — при успехе → CLOSED, при ошибке → OPEN |

Параметры: `FAILURE_THRESHOLD=3`, `RECOVERY_TIMEOUT=60s`, `SUCCESS_THRESHOLD=1`.

---

## Rate Limiting

Если провайдер зарегистрирован в `ProviderRegistry` с `rate_limit_rpm > 0`, балансировщик отслеживает количество запросов в скользящем окне 60 секунд. При превышении лимита провайдер временно исключается из пула.

```python
# Зарегистрировать провайдер с лимитом 100 RPM
POST /providers/register
{
  "id": "openai-prod",
  "rate_limit_rpm": 100,
  ...
}
```

## Сравнение стратегий

| Критерий | round_robin | weighted | latency_based |
|---|---|---|---|
| Равномерность запросов | ✅ Идеальная | ✅ Пропорциональная | ❌ Перекос к быстрым |
| Учёт скорости | ❌ Нет | ❌ Нет | ✅ Автоматически |
| Учёт приоритета | ❌ Нет | ✅ Через weight | ✅ Через priority |
| Предсказуемость | ✅ Высокая | ✅ Высокая | ⚠ Зависит от нагрузки |
| Overhead | ✅ Минимальный | ✅ Минимальный | ⚠ EMA-расчёт |

**Рекомендация:** Для production с разнородными провайдерами — `latency_based` + настройка `priority` через реестр. Для простых сценариев с мок-серверами — `round_robin`.

---

## Failover

При любой ошибке провайдера балансировщик автоматически пробует следующий доступный провайдер (до `MAX_RETRIES=2` повторных попыток). Tried-провайдеры исключаются из выборки.
